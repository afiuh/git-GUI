# Git GUI 完整需求文档

## 1. 产品概述

一款面向开发者的 Git 图形化客户端，运行在 Web 浏览器中，通过本地后端服务与 Git 仓库交互。核心目标是提供**简洁、清晰、美观、功能全面**的操作体验，覆盖日常 Git 使用的完整流程：从克隆仓库、查看状态、暂存/提交、管理分支、查看历史到推送/拉取远程更新。

## 2. 用户角色

- 开发者：使用 Git 进行版本控制的个人或团队成员。

## 3. 核心功能模块

### 3.1 克隆仓库（Clone）

**功能描述**：首次使用或需要添加新仓库时，从远程仓库克隆代码到本地。

**界面元素**：
- 打开工具时，如果没有打开任何仓库，显示克隆向导（全屏居中卡片）。
- 已打开仓库时，可通过顶部菜单或按钮触发"克隆新仓库"。

**输入项**：
- 远程仓库 URL（支持 HTTPS/SSH）
- 本地存放路径（提供文件夹选择器，默认建议路径如 `~/Documents/<repo-name>`）

**操作流程**：
1. 用户填写 URL 和本地路径。
2. 点击"克隆"按钮。
3. 后端执行 `git clone <url> <path>`，显示克隆进度（可选，第一版可简化）。
4. 克隆成功后自动切换到该仓库，主界面显示三栏视图。

**错误处理**：
- URL 无效、网络错误、权限不足、目标路径已存在非空目录 → 显示红色错误提示。

**对应命令**：
```bash
git clone <url> <path>
```

### 3.2 变更管理（文件状态）

**功能描述**：查看工作区相对于最后一次提交的变化，控制哪些变化进入暂存区。

**界面布局**：位于中间栏。

**分组显示**（可折叠）：
- **未暂存的改动**（红色文字）
- **已暂存的改动**（绿色文字）
- **未跟踪的文件**（灰色文字）

每组内列出文件，每个文件包含：
- 文件名（使用状态颜色）
- 操作按钮：
  - 未暂存/未跟踪文件：提供"暂存"按钮（+ 图标）
  - 已暂存文件：提供"取消暂存"按钮（- 图标）
  - 所有文件提供"丢弃修改"按钮（垃圾桶图标，点击需二次确认）

**顶部工具栏**：
- 手动刷新按钮（重新加载文件状态）
- 分组折叠/展开按钮（可选）
- **批量操作**：每组提供"暂存全部"/"取消暂存全部"快捷按钮

**底部区域**：
- 提交信息输入框（多行文本）
- "提交"按钮（主要操作）

**自动刷新**：每次执行暂存、取消暂存、丢弃修改、提交、切换分支、合并等操作后，自动刷新文件列表。

**对应命令**：
- 查看状态：`git status --porcelain`
- 暂存文件：`git add <file>`
- 取消暂存：`git restore --staged <file>`
- 丢弃修改：`git restore <file>`
- 提交：`git commit -m "<message>"`

### 3.3 快照管理（提交历史）

**功能描述**：查看历史提交记录，并浏览每次提交的改动内容。

**界面布局**：位于右侧栏。

**提交列表**：
- 每行显示：短哈希（7位，等宽字体）、提交说明、作者、相对时间（如"2小时前"）。
- 默认显示最近 20 条，底部提供"加载更多"按钮。
- 当前分支的最新提交高亮（左侧蓝色竖条或浅色背景）。

**搜索过滤**：顶部搜索框支持按提交说明关键词过滤（前端过滤）。

**点击提交**：在提交列表下方展开详细信息：
- 改动文件列表（每个文件显示状态 M/A/D）
- 点击某个文件后，显示代码差异（diff）：
  - 双栏布局：左侧行号，右侧代码行
  - 添加行：绿色背景 `+` 号
  - 删除行：红色背景 `-` 号
  - 使用等宽字体 JetBrains Mono
  - 顶部显示文件路径，并提供"复制完整 diff"按钮

**对应命令**：
- 查看日志：`git log --oneline --name-status -20`
- 查看 diff：`git show <commit> -- <file>`

### 3.4 分支管理

**功能描述**：查看、创建、切换、合并、删除本地分支。

**界面布局**：位于左侧栏。

**分支列表**：
- 显示所有本地分支，当前分支加粗且有背景高亮。
- 鼠标悬停显示该分支的最新提交说明。

**操作按钮**（位于列表上方或每个分支旁）：
- **新建分支**：输入分支名，基于当前 HEAD 创建，并自动切换。
- **切换分支**：双击分支名或点击"切换"按钮。
- **合并到当前**：当前所在分支（如 main）为目标分支，选中另一个分支后点击"合并"。
- **删除分支**：仅当分支已合并且不是当前分支时允许删除。

**切换分支保护**：
- 如果当前有未提交的改动，弹出模态框提示"请先提交或暂存改动"，阻止切换。提供两个选项："暂存并切换"（自动执行 `git add .` 和切换，切换回来时这些改动仍保持暂存状态）、"强制切换（丢弃改动）"（执行 `git checkout --force`，需谨慎，第一版仅提供提示，让用户手动处理）。

**合并冲突处理**：
- 合并时若无冲突，自动完成并提示成功。
- 若产生冲突，在中间栏顶部显示警告卡片：
  - 列出冲突文件。
  - 每个冲突文件提供按钮：
    - **"使用当前分支版本"（ours）**：保留当前分支的改动，放弃被合并分支的改动。执行 `git checkout --ours <file>`。
    - **"使用被合并版本"（theirs）**：放弃当前分支的改动，保留被合并分支的改动。执行 `git checkout --theirs <file>`。
    - **"手动编辑"**：打开文件让用户自行解决冲突，编辑完成后点击"标记为已解决"。
  - 用户解决所有冲突后，点击"继续合并"按钮（执行 `git commit` 完成合并）。

**对应命令**：
- 列出分支：`git branch`
- 创建分支：`git switch -c <branch>`
- 切换分支：`git switch <branch>`
- 合并分支：`git merge <branch>`
- 删除分支：`git branch -d <branch>`
- 解决冲突后提交：`git commit`（无需 -m，因为 Git 会生成默认合并信息）

### 3.5 远程同步

**功能描述**：与远程仓库交互，拉取更新和推送本地提交。

**界面布局**：位于顶部栏右侧，或以图标按钮形式展示。

**显示信息**：
- 当前远程名称（如 `origin`）和 URL（只读）。
- 状态指示（如"已同步"或"有未拉取更新"）。

**操作按钮**：
- **拉取（Pull）**：执行 `git pull`（fetch + merge），如有冲突则触发合并冲突流程。
- **推送（Push）**：执行 `git push`。若当前分支未设置上游，自动执行 `git push -u origin <当前分支>`。若推送被拒绝（非 fast‑forward），提示"远程有新的提交，请先拉取"。

**可选按钮**：
- **获取（Fetch）**：只下载不合并，第一版可省略。

**认证**：不处理认证细节，依赖 Git 已配置的凭证管理器（SSH key 或 HTTPS 密码缓存）。若认证失败，显示错误信息并建议用户检查命令行配置。

**对应命令**：
- 拉取：`git pull`
- 推送：`git push` 或 `git push -u origin <branch>`

### 3.6 Git 命令流水线（自定义执行顺序）

**功能描述**：用户可以自定义多个 Git 命令的执行顺序，保存为"流水线"或"快捷命令"，一键执行完整流程。

**典型场景**：
- 快速提交推送：`git status` → `git add .` → `git commit -m "msg"` → `git push`
- 仅暂存并提交：`git add .` → `git commit -m "msg"`
- 仅拉取合并：`git fetch` → `git merge`

**界面元素**：
- 顶部栏或独立面板提供"流水线"入口按钮（⚡图标）。
- 点击打开流水线面板，包含：
  - **预设流水线**：提供 2-3 个常用预设（如"提交并推送"、"仅提交"）
  - **自定义流水线**：用户可添加/删除/排序命令
  - **执行按钮**：一键执行当前流水线

**流水线配置**：
- 每条流水线包含多个步骤，每个步骤包含：
  - 命令类型：下拉选择（`status`、`add`、`commit`、`push`、`pull`、`fetch`、`merge` 等）
  - 参数配置：
    - `add`：指定文件路径（`.` 表示全部）
    - `commit`：指定提交信息
    - 其他命令可留空使用默认值
- 支持拖拽排序调整执行顺序
- 支持启用/禁用某个步骤（跳过但不删除）

**流水线管理**：
- 支持创建多条流水线，命名保存
- 支持编辑、删除已有流水线
- `localStorage` 持久化存储流水线配置

**执行逻辑**：
1. 按顺序依次执行每个启用的步骤
2. 前一步失败时停止执行，显示错误信息
3. 显示执行进度（当前步骤/总步骤）
4. 全部成功后显示成功提示

**错误处理**：
- 单个步骤失败时：
  - 停止后续步骤
  - 显示失败步骤的命令和错误信息
  - 提供"重试当前步骤"和"查看详情"按钮
- 支持中断执行（取消按钮）

**对应命令**（按步骤实际执行的命令）：
```bash
# 示例：提交并推送流水线
git status
git add .
git commit -m "提交信息"
git push
```

## 5. 交互与反馈规范

- **异步操作**：提交、推送、拉取、合并、克隆等显示顶部进度条（线性进度条，位于顶部栏下方），同时禁用相关按钮。
- **加载状态**：文件列表加载时显示骨架屏（skeleton）；按钮禁用时显示 loading spinner。
- **成功反馈**：右下角 toast 提示（绿色边框，3秒消失）。
- **错误反馈**：红色边框弹窗，显示原始 Git 错误信息（可折叠详情），提供"复制错误"按钮。
- **空状态**：当没有改动时，中间栏显示插画和文字："工作区干净，没有什么需要提交的"。
- **并发操作保护**：提交、暂存等操作完成后才能执行下一次操作，防止重复提交。

## 6. 技术实现建议

### 6.1 架构
- **前端**：Vite + 原生 JavaScript（简化复杂度）
- **后端**：Node.js + Express + simple-git（REST API）
- **通信**：Vite 开发服务器（端口 5173）代理 `/api/*` 到后端（端口 3001）
- **启动方式**：用户双击 `start.bat`，同时启动后端服务和 Vite 开发服务器，自动打开 `http://localhost:5173`

### 6.2 后端 API 设计（REST）

**统一响应格式**：
```json
{
  "code": 0,
  "message": "success",
  "data": null
}
```
- `code: 0` = 成功，非 0 = 错误
- `message`: 成功时为 "success"，失败时为错误描述
- `data`: 具体返回数据，失败时为 null

**超时机制**：
- 所有 Git 命令执行超时时间：**60 秒**
- 超时后返回错误："Git 命令执行超时，请检查仓库状态"

**API 端点**：

| 端点 | 方法 | 功能 | 响应 data 结构 |
|------|------|------|----------------|
| `/api/clone` | POST | 克隆仓库（body: `{ url, path }`） | `{ path: string }` |
| `/api/status` | GET | 返回 `git status --porcelain` 解析结果 | `{ modified: [], staged: [], untracked: [] }` |
| `/api/add` | POST | 暂存文件（body: `{ files: string[] }`） | `null` |
| `/api/restore` | POST | 取消暂存或丢弃修改（body: `{ files: string[], staged: boolean }`） | `null` |
| `/api/commit` | POST | 提交（body: `{ message: string }`） | `{ hash: string }` |
| `/api/log` | GET | 获取提交历史（参数 `?n=20&search=keyword`） | `[{ hash, message, author, date }]` |
| `/api/show` | GET | 获取某提交的某文件 diff（参数 `?commit=<hash>&file=<path>`） | `{ diff: string }` |
| `/api/branches` | GET | 获取本地分支列表 | `{ current: string, all: string[] }` |
| `/api/branch/new` | POST | 创建分支（body: `{ name: string }`） | `{ name: string }` |
| `/api/branch/switch` | POST | 切换分支（body: `{ name: string }`） | `null` |
| `/api/branch/merge` | POST | 合并分支（body: `{ fromBranch: string }`） | `{ hasConflict: boolean, conflictFiles: [] }` |
| `/api/branch/delete` | DELETE | 删除分支（body: `{ name: string }`） | `null` |
| `/api/pull` | POST | 拉取远程更新 | `{ hasConflict: boolean }` |
| `/api/push` | POST | 推送当前分支 | `null` |
| `/api/repo/info` | GET | 返回当前仓库信息 | `{ path, remote, currentBranch, isRepo }` |
| `/api/pipeline/execute` | POST | 执行流水线（body: `{ steps: [{ command, args }] }`） | `{ results: [{ success, output, error }] }` |

### 6.3 前端持久化

**仓库路径存储**：
- 使用 `localStorage` 存储：最近打开的仓库路径（`lastRepoPath`）
- 用户打开工具时，自动加载上次仓库（如果存在且仍为有效 Git 仓库）
- 首次使用时，显示克隆向导

**主题偏好**：
- 使用 `localStorage` 存储：`theme`（`light` | `dark`）
- 页面加载时应用保存的主题

**流水线配置**：
- 使用 `localStorage` 存储：`pipelines`（JSON 数组）
- 每条流水线结构：`{ id, name, steps: [{ command, args, enabled }] }`
- 预设 2-3 条常用流水线供首次使用

## 7. 非功能性需求

- **性能**：对于大型仓库（10 万+文件），`git status` 可能较慢，需显示 loading 动画，避免界面假死。
- **安全性**：后端只允许访问本地文件系统，不暴露给外部网络（绑定 `localhost`）。
- **错误恢复**：如果 Git 命令超时或意外退出，应给出明确提示，并允许用户重试。
- **可扩展性**：模块化设计，便于后续增加 stash、rebase、标签管理等功能。
